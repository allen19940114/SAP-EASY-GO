# OLORA 系统测试计划

## 测试概述

**项目名称**: OLORA (Operational Intelligence & Reporting Assistant)
**测试范围**: 前端功能、后端API、数据库交互
**测试类型**: 功能测试、集成测试、端到端测试
**测试环境**:
- 前端: http://localhost:3001
- 后端: http://localhost:3002
- 数据库: PostgreSQL (localhost:5432)

---

## 测试目标

1. **功能完整性验证**: 确保所有功能模块按照架构设计正常工作
2. **数据完整性验证**: 确保前端操作能正确写入、查询、修改数据库数据
3. **用户体验验证**: 确保界面友好、交互流畅、错误提示清晰
4. **安全性验证**: 确保认证授权机制有效、敏感数据保护
5. **性能验证**: 确保响应时间合理、无明显卡顿

## 测试策略

### 1. 单元测试
- **工具**: Jest
- **范围**: 后端 Service、Controller、Utility 函数
- **覆盖率目标**: ≥ 80%

### 2. 前端组件测试
- **工具**: React Testing Library
- **范围**: UI 组件、交互逻辑
- **覆盖率目标**: ≥ 70%

### 3. 端到端测试 (E2E)
- **工具**: Playwright
- **范围**: 完整用户流程
- **关键场景**:
  - 注册→登录→对话→生成报表→登出
  - 上传知识库→对话引用知识→验证准确性

### 4. 数据库集成测试
- **工具**: Prisma + Jest
- **验证点**:
  - CRUD 操作正确性
  - 事务一致性
  - 关联查询
  - 数据迁移

### 5. API 集成测试
- **工具**: Supertest + Jest
- **验证点**:
  - 接口响应格式
  - 错误处理
  - 认证授权
  - 参数验证

## 测试用例设计原则

### 正向测试
- 正常输入、正常流程
- 验证功能符合预期

### 负向测试
- 异常输入 (空值、超长、特殊字符)
- 边界值测试
- 错误处理验证

### 安全测试
- SQL 注入防护
- XSS 防护
- CSRF 防护
- 权限控制验证

### 性能测试
- 响应时间 < 2秒
- 并发用户 100+
- 大文件上传 (< 100MB)
- 长对话会话 (100+ 消息)

## 缺陷管理

### 严重等级定义
- **P0 (Blocker)**: 系统崩溃、数据丢失、无法使用核心功能
- **P1 (Critical)**: 主要功能异常、数据错误
- **P2 (Major)**: 次要功能异常、体验问题
- **P3 (Minor)**: UI 细节、文案错误

### 缺陷处理流程
1. 发现缺陷 → 记录到 `05_BUG记录及修复方式.md`
2. 分析根因 → 定位代码位置
3. 修复代码 → 本地验证
4. 回归测试 → 确认修复
5. 更新测试结果 → 标记为已修复
